02 / Le problème

Le prop drilling

Les données passent de composant en composant via props, comme un seau d'eau qu'on se passe de main en main.

App pokemons = [...]
↓ :pokemons
PageAccueil
↓ :pokemons
PokemonCard

Et si PageDetail a aussi besoin des données ? Il faut tout remonter dans App et redescendre...

App Doit stocker TOUTES les données
↓
PageAccueil
↓
PokemonCard
↓
PageDetail

⚠️ Problème

C'est comme si chaque pièce de la maison avait sa propre réserve de nourriture au lieu d'un frigo central.

04 / Anatomie

Les 3 parties d'un store

📦

State

Les données

Comme ref() mais partagé entre tous les composants

pokemons, isLoading
🧮

Getters

Données calculées

Comme computed() mais dans le store

totalPokemons
⚡

Actions

Fonctions

Modifient le state (sync ou async)

fetchPokemons()

💡 Analogie

Le State c'est le frigo, les Getters sont les recettes, et les Actions sont le cuisinier.

05 / Créer un store

Créer un store

📄 stores/pokemonStore.js

import { defineStore } from 'pinia'

export const usePokemonStore =
  defineStore('pokemon', {

  // 📦 State — les données
  state: () => ({
    pokemons: [],
    isLoading: false,
    error: null,
  }),

  // 🧮 Getters — données calculées
  getters: {
    totalPokemons: (state) =>
      state.pokemons.length,

    getPokemonById: (state) => {
      return (id) =>
        state.pokemons.find(
          p => p.id === id
        )
    },
  },

  // ⚡ Actions — fonctions
  actions: {
    async fetchPokemons() {
      this.isLoading = true
      this.error = null
      try {
        const res = await fetch(
          'http://localhost:3535/pokemons'
        )
        this.pokemons =
          await res.json()
      } catch (err) {
        this.error =
          'Impossible de charger'
      } finally {
        this.isLoading = false
      }
    },
  },
})

1. defineStore

Crée le store avec un identifiant unique ('pokemon') et un objet de configuration.

2. Convention de nom

Le nom commence par use et finit par Store : usePokemonStore

3. state est une fonction

Le state retourne un objet (comme data() en Options API). Chaque propriété doit être initialisée.

4. this dans les actions

Les actions accèdent au state via this (pas besoin de .value).

06 / State

Le state — les données

Le state est une fonction qui retourne un objet contenant toutes les données du store.

state: () => ({
  // Tableau de Pokémon
  pokemons: [],

  // Indicateur de chargement
  isLoading: false,

  // Message d'erreur
  error: null,

  // Filtre de recherche
  searchQuery: '',
})

✓ Toujours initialiser

Chaque propriété doit avoir une valeur par défaut : tableau vide [], booléen false, ou null.

✓ Réactif automatiquement

Pinia rend le state réactif — les composants qui l'utilisent se mettent à jour automatiquement.

✓ Partagé partout

Tous les composants accèdent aux mêmes données. Plus besoin de props pour passer les données.

⚠️ Règle importante

Ne jamais modifier le state directement depuis un composant — toujours passer par une action du store.

07 / Getters

Les getters — données calculées

Getter simple

S'utilise comme une propriété (sans parenthèses)

// Définition
getters: {
  totalPokemons: (state) =>
    state.pokemons.length,
}
// Utilisation dans le template
<p>{{ store.totalPokemons }} Pokémon</p>

Getter avec paramètre

Retourne une fonction qu'on appelle avec un argument

// Définition
getters: {
  getPokemonById: (state) => {
    return (id) =>
      state.pokemons.find(
        p => p.id === id
      )
  },
}
// Utilisation
const pikachu =
  store.getPokemonById(25)

💡 Réactivité automatique

Les getters se recalculent automatiquement quand le state change — exactement comme computed().

08 / Actions

Les actions — modifier le state

Les actions sont des fonctions qui modifient le state. Elles peuvent être synchrones ou asynchrones.

actions: {
  async fetchPokemons() {
    // 1. Début du chargement
    this.isLoading = true
    this.error = null

    try {
      // 2. Appel API
      const res = await fetch(
        'http://localhost:3535/pokemons'
      )
      // 3. Stocker les données
      this.pokemons =
        await res.json()

    } catch (err) {
      // 4. Gérer l'erreur
      this.error =
        'Impossible de charger'

    } finally {
      // 5. Fin du chargement
      this.isLoading = false
    }
  },
}

Pattern loading / error / data

Schéma classique pour les appels API :

1 isLoading = true
2 Appel fetch()
3 Stocker dans this.pokemons
4 Ou erreur dans this.error
5 isLoading = false

this = le store

Dans une action, this fait référence au store. Pas besoin de .value comme avec ref().

09 / Utiliser le store

Utiliser le store dans un composant

📄 pages/PageAccueil.vue

<script setup>
import { usePokemonStore }
  from '@/stores/pokemonStore'
import { storeToRefs }
  from 'pinia'
import { onMounted }
  from 'vue'

// 1. Instancier le store
const pokemonStore =
  usePokemonStore()

// 2. Extraire le state (réactif !)
const { pokemons, isLoading, error } =
  storeToRefs(pokemonStore)

// 3. Charger les données au montage
onMounted(() => {
  pokemonStore.fetchPokemons()
})
</script>

📄 template

<template>
  <div>
    <!-- Chargement -->
    <p v-if="isLoading">
      Chargement...
    </p>

    <!-- Erreur -->
    <p v-else-if="error">
      {{ error }}
    </p>

    <!-- Données -->
    <div v-else>
      <PokemonCard
        v-for="p in pokemons"
        :key="p.id"
        :pokemon="p"
      />
    </div>
  </div>
</template>

💡 Remarquez

Grâce à storeToRefs, on utilise pokemons directement dans le template, sans store.pokemons.

10 / storeToRefs

storeToRefs — pourquoi ?

❌ Sans storeToRefs 

const store = usePokemonStore()

// ❌ PERD la réactivité !
const { pokemons } = store

// pokemons est une copie figée
// Le template ne se met plus à jour

La destructuration casse le lien réactif avec le store. La variable pokemons est une simple valeur, pas une ref.

✅ Avec storeToRefs 

import { storeToRefs } from 'pinia'

const store = usePokemonStore()

// ✅ GARDE la réactivité !
const { pokemons } =
  storeToRefs(store)

// pokemons est une ref réactive

Chaque propriété destructurée reste une ref réactive. Le template se met à jour quand le store change.

La règle à retenir

📦

State et Getters

→ storeToRefs(store)

pokemons, isLoading, totalPokemons

⚡

Actions

→ store.fetchPokemons()

Appel direct sur le store

⚠️ Erreur la plus fréquente avec Pinia

Oublier storeToRefs est la source de bugs #1 chez les débutants. Si votre template ne se met pas à jour, vérifiez ça en premier !

11 / Quand utiliser

Quand utiliser un store ?

✅ Store

• Données partagées entre plusieurs pages
• État global : authentification, panier, favoris
• Données d'API utilisées par plusieurs composants

✅ Props / Emits

• Données locales à un composant et son enfant direct
• Communication parent ↔ enfant simple
• Données qui ne sortent pas du composant

❌ Pas un store

• État d'un formulaire
• Toggle d'un menu / modal
• État UI temporaire (hover, focus...)

Règle d'or

Si 2+ composants non-liés ont besoin de la même donnée → store
